## Step 1: v1 → v2 auto-migration（5 分）

v0.2 の `loadConfig` は schema=1 を検出すると自動で v2 に書き換える。**実 write が走るのでバックアップが取れていることを再確認してから次の手順に進む**。

```bash
# 1. dry-run でプレビュー（書き込みなし）
aiftp config migrate --dry-run
```

期待される出力：

```
Dry-run preview: schema = 1 -> schema = 2

[encoding]
file_name = "auto"

[quirks]
ignore_pasv_address = false
use_mlsd = true
tls_check_hostname = true
noop_interval_sec = 0
```

```bash
# 2. 実 migration（自動で .aiftp.toml.v1.bak が作られる）
aiftp config migrate
```

期待される出力：

```
Migrated .aiftp.toml from schema 1 to schema 2.
Original preserved as .aiftp.toml.v1.bak
```

**確認**：

```bash
diff .aiftp.toml.v1.bak .aiftp.toml.pre-v0.2.bak   # 完全一致のはず
head -1 .aiftp.toml                                # schema = 2
ls .aiftp/logs/                                    # migrations.jsonl が生成されている
cat .aiftp/logs/migrations.jsonl                   # fromSchema=1 / toSchema=2 ログ
```

**回帰検証**：

```bash
aiftp status --profile production    # v0.1.1 と同じ diff が返るはず
aiftp push --dry-run --profile production
```

挙動に差分がないことを確認。差分があったら **そこで止めて報告**してください。

## Step 2: 自動 migration 多重実行の安全弁（2 分）

`.v1.bak` が既に存在するときの保護：

```bash
# 既に bak があるので、再 migration を試みると拒否されるはず
aiftp config migrate
```

期待される出力：

```
Already at latest schema (schema = 2)
```

（schema=2 になっているので migrate 不要、no-op。`.v1.bak` の上書きは起きない）

念のため、 `.v1.bak` を削除してから手で `schema = 1` に書き戻して、もう一度試して **本物の guard** を確認することも可能（任意）：

```bash
# 任意の追加検証（時間があれば）
cp .aiftp.toml.v1.bak .aiftp.toml
# 既存 .v1.bak がある状態で migration を呼ぶ -> 拒否
aiftp config migrate
# 期待: ConfigError "Refusing to migrate because ... .v1.bak already exists"
# 終わったら：
mv .aiftp.toml.v1.bak .aiftp.toml
```

## Step 4: `aiftp import filezilla` 動作確認（10 分）

実 FFFTP 環境がない場合は合成 XML で代用：

```bash
# 合成 XML を作る（小さく確認用）
mkdir -p /tmp/aiftp-import-test
cat > /tmp/aiftp-import-test/sites.xml <<'XML'
<?xml version="1.0" encoding="UTF-8"?>
<FileZilla3 version="3.66.0">
  <Servers>
    <Server>
      <Host>ftp.example.test</Host>
      <Port>21</Port>
      <Protocol>4</Protocol>
      <User>imported-deploy</User>
      <Pass encoding="base64">aW1wb3J0LXNlY3JldA==</Pass>
      <Logontype>1</Logontype>
      <Name>Imported Test Site</Name>
      <RemoteDir>1 0 11 public_html</RemoteDir>
    </Server>
  </Servers>
</FileZilla3>
XML

# Dry-run で内容確認
cd ~/Projects/Web/GWco
aiftp import filezilla /tmp/aiftp-import-test/sites.xml --dry-run
```

期待される出力：

```
dry-run mode: would import 1 profile(s), skipped 0
would create [profile.imported-test-site] host=ftp.example.test user=imported-deploy password=***
```

**実 import**:

```bash
aiftp import filezilla /tmp/aiftp-import-test/sites.xml
```

期待される出力：

```
imported 1 profile(s), skipped 0, warnings 0
```

**確認**：

```bash
grep "imported-test-site" .aiftp.toml                  # 追加されている
grep "password\|cGFzcw==\|aW1wb3J0" .aiftp.toml         # **何もヒットしないこと**（password は TOML に書かない）
security find-generic-password -s "aiftp:imported:imported-test-site" -a imported-deploy -w 2>/dev/null
                                                        # -> "import-secret" が返る（Keychain に保存されている）
```

最後の `security` コマンドは macOS でのみ動作。Touch ID が反応するはず。

**クリーンアップ**：

```bash
# 取り込んだテスト profile を削除（手動）
# .aiftp.toml から [profile.imported-test-site] ブロックを削除
# Keychain からも削除
security delete-generic-password -s "aiftp:imported:imported-test-site" -a imported-deploy
rm -rf /tmp/aiftp-import-test
```

## Step 5: `aiftp profile export filezilla` 動作確認（5 分）

```bash
cd ~/Projects/Web/GWco
aiftp profile export filezilla -o exported.xml
```

期待される出力：

```

```

**確認**：

```bash
# パスワードが含まれていないこと
grep -i "pass\|secret" exported.xml | head -5
# 期待: <Pass></Pass> のような空の Pass 行のみ

xmllint --noout exported.xml && echo "valid XML"
```


## Step 6: MCP プレビュー/確認の二段流れ（10 分・任意）

Claude Code or 別の MCP クライアントから aiftp MCP に接続して確認。

```bash
# Terminal 1: MCP サーバ起動
aiftp mcp
```

Claude Code の `mcpServers` 設定例：

```json
{
  "mcpServers": {
    "aiftp-gwco": {
      "command": "node",
      "args": [
        "/Users/ytanaka/Projects/Web/AIftp/aiftp/packages/cli/dist/bin.js",
        "mcp"
      ],
      "cwd": "/Users/ytanaka/Projects/Web/GWco"
    }
  }
}
```

Claude Code を再起動して `/mcp` で aiftp-gwco が登録されているか確認。

**確認シーケンス**：

1. AI に `aiftp_status` を実行させる → diff が JSON で返る
2. AI に `aiftp_push` を `dry_run: false` で実行依頼してみる → **拒否される**こと（`aiftp_push_prepare + aiftp_push_confirm を使え` というメッセージ）
3. AI に `aiftp_push_prepare` を呼ばせる → plan_id / diff_hash / confirm_token が返る
4. AI に `aiftp_push_confirm` でその plan_id を渡させる → 実 push が走る
5. `aiftp_push_confirm` を **同じ plan_id** で再度呼ばせる → `Unknown or expired plan_id` で拒否される
6. `aiftp://config` リソースを読ませる → host / user / keychain_service が含まれない JSON が返る

注意：MCP 経由の実 push は **本番に書き込みが走る**。テスト用サブディレクトリを `remote_root` に持つ別 profile を作っておくと安全。
