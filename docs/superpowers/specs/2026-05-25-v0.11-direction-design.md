## 0. なぜ保留中か

本設計は v0.10.4 リリース直後（2026-05-25 朝）のブレインストーミングで以下のように仮決定された:

- **主目的**: Adoption / Growth
- **ターゲット**: 日本の FFFTP / FileZilla 卒業層
- **サイズ**: Large (3-4 週、6 月下旬〜7 月 ship)
- **アプローチ**: 「FFFTP 卒業 Storyline」(4 ピラー × 3 週 + マーケ仕上げ)

しかし田中さんから 2 点の本質的指摘があり保留:

1. **「FFFTP は昭和」は事実誤認** — FFFTP 初版は 1997 年（平成 9 年）公開。Zenn 記事案を訂正必須。
2. **WordPress 利用者への訴求が薄い** — Computer Use / Claude in Chrome / Operator が WordPress 管理画面をブラウザ操作できる時代に、aiftp の差別化が時限式（「今は Computer Use が遅いから」）に依存している。5 年スパンで残る差別化を立てないと、Large サイズの投資は危険。

## 1. プロダクトコンセプト（再確認）

aiftp = AI エージェントが書いたコードを FTP/FTPS サーバへ自動かつ安全にデプロイする開発者ツール（v0.10.4 時点で confirmed）。

差別化の核心仮説:

| シナリオ | Computer Use / Browser Use | aiftp |
|---|---|---|
| WordPress 投稿 1 件作成 | ✅ ブラウザで管理画面 | ❌ 守備範囲外 |
| テーマファイル 1 個編集 | ⚠️ テーマエディタで可能だが拡張性なし | ✅ ローカル編集 → `aiftp push` |
| プラグイン ZIP UP | ✅ 管理画面から可 | ✅ 同等（差なし） |
| 100 ファイル一括 UP | ❌ ブラウザ 10-20 分 | ✅ 数十秒、diff のみ |
| ローカルテスト → 本番反映 | ❌ 差分管理できない | ✅ 標準ワークフロー |
| 誤デプロイの即時ロールバック | ❌ 戻すボタン無し | ✅ 暗号化バックアップから 1 コマンド |
| `.env` / `wp-config.php` 誤 UP 防止 | ❌ ガード無し | ✅ hard-exclude で構造的 |
| AI agent の本番上書き防止 | ❌ ガード無し | ✅ prepare → confirm 2 段ゲート |
| 複数サイト横断管理 | ❌ 1 サイトずつログイン | ✅ multi-profile + AIFTP_PROFILE |

→ **暫定結論**: aiftp が刺さるのは「WordPress 投稿者」ではなく「**WordPress テーマ / プラグイン開発者 + 静的サイト / Jamstack を FTP デプロイする層**」。ただし Computer Use 成熟による時限式優位性のリスクあり。

## 6. ピラー α 詳細 — init UX framework

### 設計: `PromptFlow` ステートマシン

`packages/cli/src/prompt-framework/prompt-flow.ts` (新規 ~250 行)

```ts
export interface PromptField<T = string> {
  name: string;
  label: string;
  type: 'text' | 'password' | 'number' | 'select' | 'confirm';
  hint?: string;        // A: 入力ヒント (説明文)
  example?: string;     // A: 具体例
  validate?: (value: T, answers: Record<string, unknown>) => true | string;
  sanitize?: (raw: unknown) => T;
  initial?: T | ((answers: Record<string, unknown>) => T);
}

export class PromptFlow {
  async run(): Promise<{ kind: 'completed'; answers } | { kind: 'cancelled' }> {
    let cursor = 0;
    const answers: Record<string, unknown> = {};
    while (cursor < this.fields.length) {
      const field = this.fields[cursor];
      this.printHint(field);                              // A
      const result = await this.askField(field, answers);
      if (result.kind === 'value')   { answers[field.name] = result.value; cursor++; }
      else if (result.kind === 'back') { cursor = Math.max(0, cursor - 1); }  // B
      else if (result.kind === 'cancel') return { kind: 'cancelled' };
    }
    return { kind: 'completed', answers };
  }
}
```

### A: 入力ヒント

```
? FTP/FTPS port
  💡 標準ポート: 21 (FTP), 990 (FTPS implicit)。それ以外は確認画面が出ます。
  例: 21
 › _
```

v0.10.3 の事後確認 → 事前ヒントに格上げ。

### B: 戻りナビ (`:back` 予約キーワード)

```
? FTP host
  💡 サーバから渡されたホスト名 (例: ftp.lolipop.jp)
 › :back
↩ 前のフィールド (FTP profile name) に戻ります
```

- `:back` は sanitize 前にチェック（control char reject より優先）
- 1 問目で `:back` → no-op + 「もう戻れません」表示
- summary review (v0.10.4 C-leg) も `:back` 対応
- 数値プロンプトでも `:back` 文字列受理 → 内部で type 判定

## 8. ピラー γ 詳細 — SFTP 対応（v0.11 最重要）

### 必要性（リサーチ根拠）

競合 **alxspiker/mcp-server-ftp** が既に FTP/FTPS/**SFTP** に対応している。aiftp は v0.10.4 時点で FTP/FTPS のみ。差別化逆転を防ぐため v0.11 で SFTP 対応必須。

加えてターゲット層の中:
- 制作会社（中堅以上）は VPS / 専用サーバー利用率が高い
- WordPress フリーランスでも ConoHa VPS / さくら VPS を使う層あり
- これらは全て SSH/SFTP 環境

### 採用ライブラリ

**`ssh2-sftp-client`** （npm `ssh2-sftp-client@latest`）

理由:
- `ssh2` (Node.js SSH 実装の標準) のラッパー、active development
- TypeScript types 提供（`@types/ssh2-sftp-client`）
- API が `basic-ftp` (現 FTP/FTPS 実装) と類似 → アダプター実装が楽
- パスワード認証 + SSH 鍵認証両対応
- macOS / Linux / Windows 全動作

### 設計

`packages/core/src/sftp-client.ts` (新規 ~300 行) — 既存 `ftp-client.ts` と**同じインターフェース**を実装する SFTP アダプター。

```ts
// 既存
import { FtpClient } from './ftp-client.ts';  // basic-ftp 経由

// 新規
import { SftpClient } from './sftp-client.ts';  // ssh2-sftp-client 経由

// 上位層は protocol 別に factory で選択
function createDeployClient(profile: Profile): DeployClient {
  if (profile.protocol === 'sftp') return new SftpClient(profile);
  return new FtpClient(profile);  // ftp / ftps はこちら
}
```

### `.aiftp.toml` 拡張

```toml
[profile.production]
protocol = "sftp"           # 既存: "ftp" | "ftps"。新規: "sftp" 追加
host = "vps.example.com"
port = 22                   # 既存 (デフォルト変わる: sftp なら 22, ftp なら 21, ftps なら 21/990)
user = "deploy"
remote_root = "/var/www/html"

# SSH 鍵認証 (任意)
ssh_key_path = "~/.ssh/id_ed25519"
# パスワード認証なら従来通り Keychain に格納
```

**Hard-exclude は SFTP でも同じ**（`.env`, `wp-config.php`, `*.pem`, `db.php` は protocol によらず保護）。

### doctor 拡張

`aiftp doctor` に SFTP 用 check 追加:
- `ssh-port-reachable` (port 22 到達性)
- `ssh-key-permissions` (鍵ファイル 600/400 確認)
- `sftp-handshake` (SSH ハンドシェイク成功)
- `sftp-remote-root` (remote_root 到達性、`stat`)

### import 拡張

FileZilla `sitemanager.xml` の `Protocol=1` (SFTP) を import 対象に含める（v0.10 までは skip）。

### 実装範囲

| ファイル | 行数 | 内容 |
|---|---|---|
| `packages/core/src/sftp-client.ts` | ~300 | SFTP アダプター |
| `packages/core/src/deploy-client-factory.ts` | ~50 | protocol 別 factory |
| `packages/core/src/config.ts` | +40 | zod schema に `sftp` + `ssh_key_path` 追加 |
| `packages/core/src/doctor.ts` | +100 | SFTP 用 4 check 追加 |
| `packages/core/src/import-filezilla.ts` | +30 | SFTP profile 対応 |
| `packages/core/src/sftp-client.spec.ts` | ~400 | SFTP mock テスト |
| `packages/core/src/doctor.spec.ts` | +20 tests | 新 check |

### MCP 仕様

既存 MCP tools（aiftp_push, aiftp_status 等）は protocol 透過 — SFTP profile でもそのまま動作。新 tool 追加なし。

### smoke test CI（GitHub Actions）

`.github/workflows/smoke.yml`:

```yaml
name: smoke
on:
  release: { types: [published] }
  workflow_dispatch: {}
  schedule:
    - cron: '0 0 * * 1'   # 月曜 09:00 JST

jobs:
  smoke:
    strategy:
      fail-fast: false
      matrix:
        os: [macos-latest, ubuntu-latest, windows-latest]
        node: [22, 24]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/setup-node@v4
        with: { node-version: ${{ matrix.node }} }
      - run: npm install -g @aiftp-tools/cli@latest
      - run: aiftp --version
      - name: doctor local-only
        run: mkdir -p /tmp/smoke && cd /tmp/smoke && aiftp doctor --json || true
      - name: init non-TTY guard
        run: set +e; aiftp init < /dev/null; test $? -ne 0
      - name: MCP smoke
        run: bash ./.github/workflows/scripts/mcp-smoke.sh
```

## 5. リサーチ結果サマリ（2026-05-25 田中さん Gemini Deep Research 実施）

出典: Gemini Deep Research レポート「Claude Code を活用した WordPress 開発デプロイ自動化と日本市場におけるホスティング・保守管理の技術革新」（参考文献 8 件付き）

### 🔴 当初仮説の根本的修正が必要な発見

#### A. WordPress 国内シェア = **83.2%**（前年比 +0.3%）

当初仮説では「WordPress は減少傾向、Jamstack へ移行中」と想定したが、**実態は逆**。日本市場では WordPress は**加速継続中**（前年比 +0.3%）。Wix はわずか 0.2%、Studio (国内クラウド CMS) でも 1.6%。**WordPress を捨てる戦略は誤り**。

#### B. ホスティング上位 3 社で 51.8% / 全 GMO 系合計でさらに高シェア

- エックスサーバー 31.7% / さくら 10.7% / ロリポップ 9.4%
- GMO 系（ロリポップ / ConoHa WING / お名前.com / heteml / コアサーバー）合計が国内 1/4 以上
- これら全て **SSH/Git-WebHook/コンテナ非対応**（公式リサーチで明示）
- → aiftp の本質的価値: **SSH/WP-CLI が使えない 50%+ のサーバーで AI 連携できる唯一の道**

#### C. AI agent + FTP の競合は**既に存在**

| プロジェクト | 言語 | プロトコル | 暗号化 | 機能 |
|---|---|---|---|---|
| **alxspiker / mcp-server-ftp** ⚠️ | TypeScript | FTP/FTPS/SFTP | AES-256-GCM | フル CRUD |
| filepulse / sftp-mcp-server | Python | SFTP only | プレーン | フル |
| CDataSoftware / sftp-mcp-server | Java | SFTP | JDBC | 読取のみ（有料）|
| gradyyoung / sftp-ssh-mcp | TypeScript | SFTP+SSH | パス/SSH 鍵 | フル+SSH 実行 |
| veithly / ssh-client-mcp | Node.js | SFTP+SSH | セッション制限 | フル |

**alxspiker/mcp-server-ftp が aiftp と最も競合**。TypeScript / AES-256-GCM / FTP/FTPS まで同じ。**SFTP 対応の点でアドバンテージあり**。

#### D. WordPress 公式 MCP は WP コア側で動いている

- **Official WP MCP Adapter** (Abilities API, WP 6.7+) — WordPress コアチームが MCP 化を進めている
- AI Engine (Meow Apps), InstaWP Staging Server 等も既に存在
- → aiftp は **WordPress 機能側に踏み込むべきではない**（コア側が公式で対応するため）
- → aiftp は **FTP/FTPS 安全レイヤー**に専念すべき

#### E. WordPress 保守管理サービス市場が明確に存在

| サービス | 月額 | AI 適合性評価 |
|---|---|---|
| wp.support | 22,000円〜 | 中〜高 |
| WP保守工房 | 55,000円〜 | 中 |
| キオミル WP 保守 | 33,000円〜 | **高** |
| ビークリエイト | 13,200円〜 | **高** |
| high five create | 8,800円〜 | **高** |

→ 制作会社・保守事業者向けに「AI 連携で保守工数削減」のニーズが明確に存在
→ ターゲット選択肢 ③（制作会社・受託開発の納品自動化）の市場性が**裏付けられた**

#### F. Computer Use の WordPress 実用度（リサーチ #1 相当）

- 既に「Autonomous Autopilot」運用事例あり（Figma → AI 自動コーディング → デプロイ）
- 開発時間 90% 削減事例
- ただし役割は「ゼロからコードを書く」から「AI コードを監査・レビューする監督者」へ
- **Computer Use は WordPress 管理画面操作には有効、ただしファイル直接 UP は依然非効率**
- → aiftp との競合領域は「投稿・管理」、aiftp の領域は「テーマ・プラグイン・静的ファイル」

#### G. SWELL 等の国内独自テーマ問題

- SWELL / LIGHTNING / Cocoon 等が国内有料テーマで大シェア
- 一般 AI は「Astra」「Elementor」前提 → 国内独自テーマは知らない
- → CLAUDE.md / コンテキストファイルで補強する必要
- → aiftp のテンプレ機能に「SWELL 用」「LIGHTNING 用」を入れる価値あり

### 🎯 リサーチに基づく結論

1. **WordPress を捨てない** — 83% シェア・伸び続けている
2. **WordPress 機能側には踏み込まない** — 公式 MCP がある
3. **「SSH/WP-CLI なしで AI と連携できる唯一の deploy 基盤」が aiftp の本質的価値**
4. **競合 alxspiker と差別化必要** — 安全機構・日本特化で勝負、ただし **SFTP 未対応は v0.11 で潰すべき重要弱点**
5. **保守事業者向け B2B が大きな市場** — 月額 8,800〜55,000 円 × サイト数

### ターゲット 3 選択肢の再評価

| 選択肢 | リサーチ前 | リサーチ後 |
|---|---|---|
| ① AI agent 向け FTP 安全レイヤー | 「市場極小」 | **「市場大、明確な競合存在、差別化可能」** |
| ② Jamstack 共有サーバーデプロイ | 「層が薄い可能性」 | **「WordPress 83% で市場性弱い」** |
| ③ 日本の制作会社・受託開発の納品自動化 | 「B2B 営業必要」 | **「市場明確に存在、AI 適合性高評価」** |

→ **推奨: ① + ③ のハイブリッド** = 「日本の制作会社向け AI 連携 deploy 基盤」
