## Phase 1: 設計レベル

### 1. [CRITICAL] SFTP key auth の設計が deploy 経路に落ちていない

**ファイル**: `packages/cli/src/index.ts:892`, `packages/core/src/backup/default-store.ts:77`, `packages/mcp/src/index.ts:537`

`ssh_key_path` は schema にあるのに、実 client 作成では `password` だけを渡している。push / backup / rollback / MCP confirm で **鍵認証 SFTP が動かない**。

「schema/doctor は通るが実 deploy は鍵を使わない」という最悪状態 — schema 検証通過と実 connection 成功が乖離する。

**修正方針**: **release blocker**。`profile → CreateDeployClientOptions` の共通 builder を作り、`protocol === "sftp"` では `sshKeyPath` / passphrase を明示的に渡す。

### 2. [HIGH] α summary review と γ SFTP の組み合わせが壊れている

**ファイル**: `packages/cli/src/index.ts:402`, `packages/cli/src/index.ts:493`

SFTP が summary で `FTP` と表示され、summary edit の protocol 選択肢にも SFTP がない。Pillar α (summary review) と Pillar γ (SFTP) のクロスピラー統合が抜けていた。

**修正方針**: **release blocker**。表示を `FTP/FTPS/SFTP` 三分岐にし、edit choices に SFTP を追加。Task 31 に `:back` / summary edit + SFTP の統合テスト追加。

### 3. [HIGH] `~/.ssh/...` を受け入れる仕様だが tilde 展開がない

**ファイル**: `packages/core/src/__fixtures__/config/with-sftp.toml:11`, `packages/core/src/sftp-client.ts:65`, `packages/cli/src/index.ts:1255`

fixture と spec は `~/.ssh/id_ed25519` を許容している一方、`statSync(profile.ssh_key_path)` はそのまま読むため **ENOENT**。ドキュメント通りに書くと動かない不整合。

**修正方針**: **release blocker**。runtime 境界 (`SftpClient.connect` / `probeSftp`) で `~` を home に展開する utility を追加。fixture / spec は変更不要 (tilde 表記をそのまま許容)。

### 4. [MEDIUM] doctor 4 checks は「失敗原因の分解」が不足

**ファイル**: `packages/cli/src/index.ts:1288`, `packages/core/src/diagnostics/doctor.ts:246`

passphrase mismatch / known_hosts / remote permission denied がほぼ `sftp-handshake` or `sftp-remote-root` の文字列頼み。

**修正方針**: **next session**。`errorKind` を導入し、認証・host key・権限・remote root を分ける。

### 5. [MEDIUM] FileZilla `<Keyfile>` 未対応は明示的な skip/warn が必要

**ファイル**: `packages/core/src/importers/filezilla.ts:207`, `packages/cli/src/index.ts:1472`

parser は任意タグを `fields` に入れるが、`Keyfile` は使われず、SFTP key-only profile が password-less profile として import される。

**修正方針**: **next session**。`ssh_key_path` に変換するか、key-only SFTP は warning 付きで skip。

### 6. [LOW] 競合比較は概ね方向性 OK だが、git-ftp の SFTP 表現は要修正

**ファイル**: `docs/competitive-comparison.md:110`

git-ftp 公式は "FTP servers" を前面に出しており、SFTP は確認できなかった。`△ (実装次第)` より `× / not advertised` が安全。alxspiker は current GitHub で SFTP/FTPS 記載あり、ただし stars は 17 で「最大」表現は要注意。

**修正方針**: **v0.11.1**。

## Phase 2: 実装レベル

### 1. [CRITICAL] `createDeployClient` 呼び出しが SFTP に FTP-only props を混入し、鍵 path を落としている

**ファイル**: `packages/core/src/deploy-client-factory.ts:41`, `packages/cli/src/index.ts:892`

`protocol: profile.protocol` の object literal に `requireTls`, `verifyCertificate`, `noopIntervalSec` を渡しつつ `sshKeyPath` は渡していない。Phase 1-1 と同じ根本原因の実装側。

**修正方針**: **release blocker**。FTP/SFTP の option builder を分離し、SFTP では余計な FTP fields を渡さない。

### 2. [HIGH] SftpClient に `mapFtpError` 相当がないため backup の not-found 処理が壊れる可能性

**ファイル**: `packages/core/src/sftp-client.ts:147`, `packages/core/src/ftp-client.ts:415`, `packages/core/src/backup/default-store.ts:104`

FTP は `FtpNotFoundError` に正規化され、backup source は `null` にできる。SFTP は raw error のままなので、追加ファイルの remote source not found が backup failure になり得る。

**修正方針**: **release blocker**。`mapSftpError` を追加し、not found / auth / timeout / connection を既存 error hierarchy へ正規化。

### 3. [MEDIUM] SFTP profile export が FileZilla XML で FTP に化ける

**ファイル**: `packages/cli/src/index.ts:2525`, `packages/core/src/importers/filezilla.ts:332`

renderer は SFTP code `1` を持っているが、CLI export 側が `ftps` 以外を `ftp` に落とす。

**修正方針**: **v0.11.1**。`profile.protocol === "sftp" ? "sftp"` を追加。

### 4. [MEDIUM] Task 31 の prompt mock が本番 PromptFlow を十分に再現していない

**ファイル**: `packages/cli/src/index.spec.ts:48`, `packages/cli/src/index.spec.ts:1957`, `packages/cli/src/index.spec.ts:2025`

全 answers を毎回返すため、field-by-field の `:back`、dynamic initial、summary edit のズレを検出できない。さらに test 名は `ssh_key_path` と言っているが実際は path を入力していない。

**修正方針**: **next session**。PromptFlow 用の順次 mock を追加し、SFTP + `:back` + summary edit を通す。

### 5. [LOW] `mcp-smoke.sh` は tools/list 件数だけで initialize 成功を直接検証していない

**ファイル**: `.github/workflows/scripts/mcp-smoke.sh:20`

実用上は十分近いが、`id:1` response / `id:2` response の存在確認まで入れると smoke の説明と実体が一致する。Windows MCP probe 除外は `mktemp` / stdio / Git Bash 差分を避ける判断として妥当。

**修正方針**: **v0.11.1**。

### 6. [INFO] README リンク位置は問題なし

**ファイル**: `README.md:330`

Status / roadmap にあり、探索可能。release blocker ではない。
