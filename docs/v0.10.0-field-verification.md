> Lolipop **2026-05-30** (priority HIGH) / Xserver **2026-05-31** / Sakura **2026-06-04**.
>
> **Status**: Plan only — execution requires 田中さん explicit approval per provider, per step.

---

## 1. Goal

Verify push + prune + rollback against real Japanese shared-hosting providers before v0.10.0 release.

The goal is to confirm the four behaviors that did not exist in v0.9.x:

1. `deletion_policy = "prune-with-confirm"` triggers a real remote `delete`.
2. `deletion_policy = "prune-auto"` does the same without typed confirmation.
3. `rollback` issues a real `delete` for `added` tombstones in the target snapshot.
4. Hard-exclude paths (`wp-config.php`, `.env*`, `db.php`) are skipped from BOTH upload and delete.

## 5. Verification matrix (run per provider in order)

> Each step is run **only after the previous step's exit was 0** and 田中さん gave a verbal go.

### Step 1: Sanity

```sh
aiftp doctor --profile <provider>
```

**Pass criteria**: every probe `pass`, including `ftps-cert` and `auth`.

### Step 2: Initial push (deletion_policy = "never", default)

```sh
aiftp push --profile <provider> --dry-run
aiftp push --profile <provider>
```

**Pass criteria**:
- Dry-run reports 3 files in `Planned uploads`, 0 in `Planned deletes`.
- Real push uploads all 3 files.
- `aiftp backup list` shows one new snapshot.
- FTP client confirms the 3 files present at `remote_root`.

### Step 3: Modified push

Edit `hello-v0100.html` locally (add a single line).

```sh
aiftp push --profile <provider> --dry-run
aiftp push --profile <provider>
```

**Pass criteria**:
- Dry-run reports 1 modified file.
- New snapshot created with one entry whose `operation` is `"modified"` (verify via `aiftp backup show <id>` if available).

### Step 4: Prune verification (deletion_policy = "prune-with-confirm")

Edit `.aiftp.toml`:
```toml
[safety]
deletion_policy = "prune-with-confirm"
```

Delete `styles-v0100.css` locally (e.g. `rm styles-v0100.css`).

```sh
aiftp push --profile <provider> --dry-run
# expect: Planned deletes: 1 (styles-v0100.css)

aiftp push --profile <provider> --confirm-deletes
# expect: prompt for typed "DELETE", then proceeds
```

**Pass criteria**:
- Dry-run lists `styles-v0100.css` under planned deletes.
- Real push requires typed `DELETE` and then removes the file from remote.
- FTP client confirms file removed.
- `aiftp backup list` shows a new snapshot containing a `removed` operation for that file.

After the test, restore `.aiftp.toml`:
```toml
[safety]
deletion_policy = "never"
```

### Step 5: Rollback verification (modified → restore)

Identify the snapshot ID created in step 3 (the modified-push one).

```sh
aiftp backup list
aiftp rollback --profile <provider> --to <step3-snapshot-id> --dry-run
# expect: uploads (the pre-modified version of hello-v0100.html)

aiftp rollback --profile <provider> --to <step3-snapshot-id>
```

**Pass criteria**:
- Dry-run shows 1 upload, 0 deletes.
- Real rollback restores the pre-modified `hello-v0100.html` on the remote.

### Step 6: Rollback with delete (added → tombstone → remote delete)

This is the v0.10.0-specific case.

Identify the snapshot ID created in step 2 (the initial push).

Then push a brand new file (`new-page-v0100.html`):
```sh
echo '<html><body>new</body></html>' > new-page-v0100.html
aiftp push --profile <provider>
```

Identify the new snapshot ID (snapshot A). Now rollback to snapshot from step 5 (or any snapshot before snapshot A):

```sh
aiftp rollback --profile <provider> --to <pre-step6-snapshot-id> --dry-run
# expect: 0 uploads, 1 delete (new-page-v0100.html)
```

Confirm dry-run output matches expectation, then:

```sh
aiftp rollback --profile <provider> --to <pre-step6-snapshot-id>
# MCP path would need acknowledge_deletions: true here; CLI prompts
```

**Pass criteria**:
- Dry-run lists `new-page-v0100.html` under planned deletes.
- Real rollback removes the file from remote.
- FTP client confirms removal.

### Step 7: Hard-exclude guard

Place a sentinel file with a hard-exclude name in the test directory:
```sh
touch wp-config.php
echo '<?php define("FAKE","yes"); ?>' > wp-config.php
```

```sh
aiftp push --profile <provider> --dry-run
```

**Pass criteria**:
- `wp-config.php` is NOT in the planned uploads.
- No related warning except the standard hard-exclude notice.

Then:
```sh
rm wp-config.php
aiftp push --profile <provider> --dry-run
```

**Pass criteria**:
- `wp-config.php` is NOT in planned deletes either (hard-exclude precedes delete planning).

## 6. Safety guardrails during verification

- **NEVER** run against `glocalworks.co.jp` production `/public_html` directly.
- `remote_root` MUST be a disposable subdirectory.
- Hard-exclude check: confirm test directory has NO `wp-config.php`, `.env*`, `db.php` BEFORE step 1 (step 7 introduces and removes a sentinel).
- Always `--dry-run` before real mutation.
- Keep terminal session log (`script -aq verify-<provider>.log`) for post-mortem.
- If `aiftp` exits non-zero on step 4 or 6, STOP and report to 田中さん before continuing.

## 7. Decision tree per provider

| Outcome | Action |
|---|---|
| All 7 steps pass | Mark provider as v0.10.0-verified in `docs/compatibility-matrix.md`. |
| Step 1 (doctor) fails | Likely provider regression. Check against v0.9.5; if v0.9.5 also fails, file provider-specific issue. |
| Step 2-3 fail | Regression from v0.9.x. Halt v0.10.0 release. |
| Step 4 fails (prune) | v0.10.0 prune logic regression. Halt v0.10.0 release. |
| Step 5 fails (rollback restore) | v0.10.0 rollback regression. Halt release. |
| Step 6 fails (rollback delete) | The defining v0.10.0 feature does not work. Halt release. |
| Step 7 fails (hard-exclude) | Safety regression. Critical, halt release. |

## 9. Out of scope (post-v1.0)

- Stress test with > 100 files in a single push
- Concurrent push (multiple terminals)
- Network disruption mid-push recovery
- WordPress / live CMS targets (CMS is out of scope — `wp-config.php` is hard-excluded)
- TLS downgrade or `require_tls = false` paths (verified separately in doctor)
