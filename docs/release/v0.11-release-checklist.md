## 評価基準 (100 点満点)

aiftp の release は「テストが pass した」では不十分。
**明示的な合格ライン (90 点以上)** を超えたものだけ npm publish + tag する。

| カテゴリ | 配点 | 評価項目 | 合格ライン |
|---|---|---|---|
| **A. ユニットテスト** | 20 | pass 率 + branches coverage | pass率 ≥ 99%, branches ≥ 80% |
| **B. 統合テスト** | 15 | 4 pillar combined path | 主要ユーザーフロー 3+ シナリオ pass、prompt mock が field-by-field 再現 |
| **C. 静的検査** | 10 | lint / typecheck / format | 全クリーン |
| **D. Codex Phase 1/2 レビュー** | 20 | 各 pillar 単独 + 統合 + security の 4 ラウンド | 全 GO 判定 (CRITICAL/HIGH 残ゼロ) |
| **E. 依存 CVE** | 10 | プロダクション依存の既知脆弱性 | HIGH+CRITICAL 0 件 (devDep 含むなら別途記録) |
| **F. ドキュメント** | 10 | CHANGELOG / README / competitive-comparison / spec | 当該 release 反映済み、リンク切れなし |
| **G. 手動 smoke** | 10 | CLI / MCP の実起動確認 | `aiftp --version` / `init --help` / `doctor` / `mcp` 動作 + MCP JSON-RPC initialize |
| **H. セキュリティ非交渉ルール** | 5 | spec §B + CLAUDE.md 非交渉項目 | 全準拠 (暗号化 default ON, hard-exclude, Keychain only, .gitignore 強制) |

### スコア解釈

| スコア | 判定 |
|---|---|
| 95+ | 完全 GO、即 release 推奨 |
| 90-94 | GO、不足項目は v0.11.x backlog 記録 |
| 80-89 | GO-WITH-CAVEATS、CRITICAL/HIGH の解消必須 |
| <80 | NO-GO、不足項目を fix してから再評価 |

### スコア配点の根拠

- A は最重要だが「pass率と coverage は前提」なので 20 で打ち止め
- D は独立した目による発見が他のテストでは出ない欠陥を捉える → 20 と最高配点
- B + G は実ユーザー体験に直結するため合計 25
- E はプロダクション依存 0 件であれば満点、devDep は v0.11.1 で改善も可
- C は binary なので 10
- F + H は補完項目

### 詳細採点

#### A. ユニットテスト (19 / 20)

- pass 率: 755 / 758 = **99.6%** ✅ (合格ライン 99% 超え)
- branches coverage: **84%+** ✅ (合格ライン 80% 超え、v0.10.4 比 +1pp)
- 35+ spec files、Keychain integration test (macOS 実機)、symlink test (POSIX) 含む
- **-1 減点**: 3 skipped tests (macOS 非 CI 環境専用) は CI 設計上の制約で技術的に許容範囲

#### B. 統合テスト (9 / 15)

- `v0.11 end-to-end mock (4 pillars combined)` describe ブロックに 3 シナリオ pass:
  - `init --template wordpress-swell + protocol=sftp` round-trip
  - `init --template static` で localRoot ユーザー編集値保持 (HIGH 2 regression guard)
  - SFTP init で ssh_key_path が Keychain に漏れない
- **-6 減点**: Codex γ Phase 2-4 で「prompt mock が全 answers を毎回返すため、field-by-field の `:back` / dynamic initial / summary edit のズレを検出できない」指摘
  - v0.11.x 持ち越し: PromptFlow 用順次 mock を追加して SFTP + `:back` + summary edit 統合テスト

#### C. 静的検査 (10 / 10)

#### E. 依存 CVE (10 / 10)

`pnpm audit` 結果:

| パッケージ | 深刻度 | 場所 | プロダクション影響 |
|---|---|---|---|
| ip | HIGH (SSRF) | `packages/core` devDep > ftp-srv > ip | ❌ なし |
| esbuild | MEDIUM | root devDep > vitest > vite > esbuild | ❌ なし |
| vite | MEDIUM (Path Traversal) | root devDep > vitest > vite | ❌ なし |
| uuid | MEDIUM | `packages/core` devDep > ftp-srv > uuid | ❌ なし |

**全 4 件が devDep チェーン (`ftp-srv` / `vitest`) で、npm 公開パッケージには含まれない**。プロダクション依存 (`basic-ftp` / `ssh2-sftp-client` / `@iarna/toml` / `fast-xml-parser` / `iconv-lite` / `picomatch` / `zod` / `@modelcontextprotocol/sdk` / `commander` / `prompts`) には CVE なし。

**v0.11.1 backlog**: `ftp-srv` を別の mock (例: 自前 socket mock) に置き換えて devDep CVE を整理。

## 次回 release 時のチェックリスト操作手順

1. `cp v0.11-release-checklist.md vX.Y-release-checklist.md`
2. 採点履歴を空にする
3. v0.11.x 持ち越し項目を該当 review log に reflect
4. 4 ラウンドの Codex review を順次実施
5. 各ラウンド完了時にスコア更新
6. **90 点超え** で初めて tag + push + npm publish 進行

評価基準テーブルは原則変えない (release 間の比較性を保つため)。配点見直しは v0.12.0 / v1.0.0 などメジャー版アップ時のみ。
