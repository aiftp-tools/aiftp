# aiftp 1.0 🎉

aiftp 1.0 is the first **stable** release — built for the *制作者*
(Builder): individuals and small studios who deliver WordPress / static
sites to Japan's SSH-less shared hosting and write code with an AI agent.
After 18 incremental tags between v0.1.0 and v0.9.1, plus end-to-end
verification on Japanese shared hosting providers (see
[`docs/compatibility-matrix.md`](docs/compatibility-matrix.md) for the
honest per-provider status, including which were verified-then-cancelled),
we're confident the safety primitives hold under real conditions on the
live-verified Star Server environment.

## What aiftp is

> The safe FTP/FTPS deploy tool for the AI-agent era — built for
> Japanese shared hosting.

aiftp addresses two structural concerns that older deploy tools were
never designed for:

1. **AI agents can act on stale state or skip safety steps.** This
   failure mode has been reported publicly across major AI coding
   assistants. aiftp puts the safety guarantees (backup, hard-exclude,
   two-step confirmation) in the deploy tool itself rather than
   relying on the agent to honor them.
2. **Foreign-IP filtering on Japanese shared hosting conflicts with
   cloud CI/CD** ([Sakura's announcement since 2014-03](https://www.sakura.ad.jp/corporate/information/announcements/2014/03/10/870/)).
   When the filter is enabled for FTP/FTPS, services like Vercel /
   Netlify / Cloudflare Pages / GitHub Actions may not be able to
   complete file operations against the server. aiftp sidesteps the
   conflict by running locally from the operator's own IP — no
   provider-side security toggle needs to be weakened.

## What's in 1.0

The full feature surface as of 1.0 (mostly accumulated across
v0.1.0 – v0.9.1; see `CHANGELOG.md` for the per-version history):

### Safety primitives (the core value)

- ✅ **Two-step MCP gate** on every destructive tool:
  `aiftp_push`, `aiftp_backup_restore`, `aiftp_config_migrate`,
  `aiftp_import_filezilla`, `aiftp_import_ffftp`, `aiftp_rollback`.
  Real action requires `*_prepare` → `*_confirm` with matching
  `plan_id`, `diff_hash`, `confirm_token`. Plans expire after 5 minutes.
- ✅ **Encrypted local backup (AES-256-GCM)** of remote files,
  taken automatically before every push.
- ✅ **Two-phase atomic rollback** — staging upload + atomic rename
  per file. A mid-rollback failure can't half-roll the site.
- ✅ **Type-to-confirm production gate** for profiles matched by
  `[safety] production_profile_patterns`. No y/n the AI can skip.
- ✅ **Hard-excluded files**: `.env*`, `wp-config.php`, `*.pem`,
  `db.php`, etc. Cannot be uploaded, backed up, or restored.
- ✅ **Server-side lock file** preventing concurrent agent pushes.
- ✅ **OS Keychain credential storage**: macOS Keychain via
  `security`, Windows DPAPI via `cmdkey` + `CredRead`. Passwords
  never enter TOML, logs, or MCP responses.
- ✅ **TLS hostname mismatch is diagnosed, never silently bypassed.**

### Diagnostics & migration

- ✅ **`aiftp doctor`** — 12 diagnostic checks (config, gitignore,
  keychain, DNS, TCP, FTPS, cert chain, PASV, MLSD, SIZE,
  remote_root CWD, encoding sniff) with `--json` for AI consumption
- ✅ **`aiftp config migrate`** — v1 → v2 schema migration with
  atomic write, `.bak` preservation, multi-run guard, audit log

### Import / export / multi-profile

- ✅ **`aiftp import filezilla`** — `sitemanager.xml` with passwords
  routed to Keychain
- ✅ **`aiftp import ffftp`** — direct `ffftp.ini` parser (Shift_JIS
  via iconv-lite)
- ✅ **`aiftp profile export filezilla`** — round-trip export
- ✅ **`aiftp profile`** suite: `list / use / show / test` with
  `AIFTP_PROFILE` env override
- ✅ **MCP read-only profile tools**: `aiftp_profile_list / _current / _test`

### Watch / hook

- ✅ **`aiftp watch`** — debounced fs watcher, dry-run notifications only
- ✅ **`aiftp hook`** — Claude Code / Cursor `PostToolUse` handler,
  dry-run notifications only

### Platform support

- ✅ **macOS** verified
- ✅ **Windows 10/11** verified (DPAPI credential backend)
- 🟡 **Linux** runs but uses an in-memory credential backend (libsecret
  integration on the v1.x roadmap)

### Verified hosting providers

- ✅ **Star Server** (スターサーバー) — live-verified on real production (GWco)
- ✅ **Sakura / Lolipop / Xserver** — verified at v0.9.3 (2026-05-22); accounts since cancelled, re-verify on demand (see [ADR 0002](../adr/0002-v1.0-release-gate-redefinition.md))
- 📋 Full matrix in [`docs/compatibility-matrix.md`](docs/compatibility-matrix.md)

## What 1.0 doesn't try to do

- ❌ Auto-push on file save (defeats the safety model — use `watch`
  for notifications)
- ❌ Optional "fast mode" that disables backup or encryption
- ❌ Telemetry of any kind (see [docs/privacy.md](docs/privacy.md))
- ❌ CMS-specific staging logic (WordPress sync is out of scope —
  aiftp transports bytes)

See [`docs/roadmap.md`](docs/roadmap.md) for the full "what we will
and won't do" list.

## Getting started

```bash
# Install
npm install -g aiftp

# Initialize a project
cd ~/your-website
aiftp init

# Already use FileZilla or FFFTP? Import your sites.
aiftp import filezilla ~/.config/filezilla/sitemanager.xml
# or
aiftp import ffftp ~/Documents/ffftp.ini

# Verify the connection
aiftp doctor

# Preview what would be pushed
aiftp push --dry-run

# Real push (encrypted backup taken automatically)
aiftp push

# Something went wrong? Roll back.
aiftp rollback --steps 1
```

For the MCP / Claude Code integration:

```json
{
  "mcpServers": {
    "aiftp": {
      "command": "npx",
      "args": ["-y", "aiftp", "mcp"],
      "cwd": "/path/to/your/project"
    }
  }
}
```

The AI sees safe tools (`aiftp_status`, `aiftp_push_prepare`,
`aiftp_push_confirm`, etc.) — never the credentials, never the
ability to skip the gate.
