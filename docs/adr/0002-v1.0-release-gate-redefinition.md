# v1.0 出荷ゲートの再定義: 3社ライブ再認証を撤廃

## Decision

v1.0.0 の出荷ゲートから **「Lolipop / さくら / Xserver の実アカウントでのライブ E2E 再認証（A-7）」を撤廃**する。v1.0 は次で出荷可能とする: **(1) 保持中の StarServer（GWco）実機での E2E 検証、(2) v0.9.3（2026-05-22）時点で記録済みの3社検証結果、(3) ftp-srv モックによる smoke CI**。未消化の delta（最小 push の再確認・Shift_JIS 挙動）は、需要発生時——特に **デプロイ代行案件で実際に客のサーバへ触れた機会**——に再検証する。あわせて v1.0 の位置づけを「世界へ公式推奨するマス製品」から **「制作者が実務で頼れる安定版、かつ既存サービスの信用に足るポートフォリオ」** へ再定義する（[ADR 0001](0001-product-direction-builder-wedge-services-monetization.md) と整合）。

## Context / Why

roadmap と `docs/release/v1.0.0-draft.md` は当初、v1.0 を「3社 E2E 検証完了」でゲートしていた。しかし 3社の検証は v0.9.3 で**既に完了・記録済み**であり（compatibility-matrix に doctor 11/2/0・外国IPフィルタ検証を日付付きで記録）、その**後に**3社のレンタル契約は解約された。残る delta は Shift_JIS 等のわずかな項目のみ。

ADR 0001 で「aiftp 単体を有償化せず、過剰投資しない」と決めた以上、**わずかな delta を埋めるためだけに3社を再契約するのは ROI が悪い**。一方、制作者ユースケースの実環境である StarServer は保持しており、これが最も代表性の高い検証面となる。

## Consequences

- compatibility-matrix の3社は「v0.9.3 で検証・以後アカウント解約・再検証は需要発生時」と**正直に注記**する。aiftp の信用の土台は安全性であり、検証状況を誇張すると土台が崩れるため、誇大広告にしない。
- v1.0 リリース文書のトーンを「制作者向け安定版」に調整する。
- 3社の再検証は代行案件と合流させる（収益化と検証が同じ機会に起きる）。
- 将来、実ユーザの issue や代行案件で3社のいずれかに再接続でき次第、matrix を更新する。

## Considered alternatives

- **3社を再契約してフル認証を維持** → ADR 0001 の「過剰投資しない」と矛盾。客ゼロ段階で delta を埋めるためだけの再契約は ROI が悪い。
- **v1.0 を出さず 0.x のまま据え置く** → "1.0" は制作者・サービス見込み客に対する安定性/信用シグナルとして価値がある。ポートフォリオとして 1.0 を切る意味は大きい。
