# プロダクト方向性: 楔は「制作者」、収益化は既存サービス経由、Pro 製品は据え置き

## Decision

aiftp の楔ペルソナを **「制作者（Builder）= SSH 非対応の日本のレンタルサーバへ WordPress・静的サイトを納品し、AI でコードを書く個人/小規模制作者」に1点張り**する（[CONTEXT.md](../../CONTEXT.md) 参照）。収益化は aiftp 単体のサブスク化ではなく、**aiftp を信用資産・リード獲得装置とした既存事業（デプロイ代行・コンサル・教育）経由**で行う。**Pro 製品ラインは需要が実証されるまで据え置き**、それまで **Free に拡張シーム（プラグイン機構）を仕込まない**。

## Context / Why

v0.11.0 リリース直後の npm DL は約 691/月（うち 5/24 に 497 のリリース直後スパイク＋減衰）で、**持続的な人間の利用の証拠はまだほぼ無い**。客が存在しない段階で Pro 製品（課金・ライセンス鍵・サーバ実装が必要な候補機能・サポート・プライバシーポリシー）を組むのは、roadmap 自身の「需要に先行して作らない／一人運営」方針に反する。

ニーズ×導入能力の2×2で見ると、VPS/SSH を持つ普通のエンジニアは git-ftp 等で足りニーズが薄く、真の非エンジニアは現状の導入フロア（npm/CLI/MCP/TOML/Keychain）で脱落する。**ニーズが強くかつ導入できるのは「制作者」層のみ**で、これは収益化先である田中さんのコンサル/代行/教育の客層と一致する。ツールとサービスが同じニッチを指すため、方向性として一貫する。

## Consequences

- v1.0 のゴールから「真の非エンジニアでも自力で使える」を外す（"AI エージェントが導入〜設定を全部やる" が実証できるまで努力目標）。普通のエンジニア層は最初から狙わない。
- 告知（Zenn / X）と v1.0 のスコープは、すべて「制作者」ペルソナから逆算する。
- Pro を将来出す場合は破壊的変更を許容するメジャーバージョン（2.0）で対応する。1.0 の公開 CLI/MCP 表面は小さく保つ。
- roadmap の Pro candidates は「candidates のまま塩漬け」とし、本 ADR で着手判断を需要実証後に明示的に保留する。

## Considered alternatives

- **aiftp 自体を有償サブスク製品にする** → 客ゼロのため時期尚早。育成が先。
- **Free に最小プラグインシームを 1.0 で仕込む** → 投機的。Pro が来ない可能性が高く、公開 API 表面と保守負担だけ増える。
- **マス向け汎用 FTP/デプロイツールとして普通のエンジニアも取りに行く** → git-ftp(5,582★) 等に対し差別化が薄く、勝つ必要もない。
