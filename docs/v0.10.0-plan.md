## 0. なぜ v0.10.0 が必要か

v0.9.2 で発覚した致命的ギャップ（CHANGELOG.md `[0.9.2]` Known limitations 第 1 項）：

- **added-only push の rollback が remote から delete しない**（snapshot は metadata-only として作られるが内容空、rollback は no-op）
- **removed（push で消えるファイル）の old version を rollback で復元できない**

→ spec の「**every push is reversible**」invariant を**真には満たしていない**状態。v0.10.0 で完全実装する。

---

## 1. 現状（v0.9.3 時点）の問題

### 1.1 SnapshotManifest schema 1

```ts
export interface SnapshotFileMeta {
  path: string;
  storedName: string;    // 必須 → added は何を入れる？
  sizeOriginal: number;
  // ...
}
```

問題: ファイルが「**何の操作対象だったか**」（added/modified/removed）を保持していない。

### 1.2 createAutoSnapshot

```ts
async createAutoSnapshot(files: readonly string[]): Promise<SnapshotMeta>
```

問題: path のリストしか受け取らないため、source.readFile の呼び出し戦略を operation 別に分岐できない。

### 1.3 rollback

```ts
for (const file of target.files) {
  // restoreFile → upload
}
```

問題: `target.files` を全部 upload するだけ。added は files に入っていないので何も起きず、removed は元から保存されない。

## 2. v0.10.0 設計

### 2.1 新 SnapshotManifest（schema 2）

```ts
const SNAPSHOT_SCHEMA = 2 as const;

export type FileOperation = 'added' | 'modified' | 'removed';

export interface SnapshotFileMeta {
  path: string;
  operation: FileOperation;            // NEW
  storedName: string | null;           // added は null
  sizeOriginal: number | null;
  sizeEncrypted: number | null;
  sha256Original: string | null;
  sha256Encrypted: string | null;
}

export interface SnapshotMeta {
  // ... 既存
  counts?: { added: number; modified: number; removed: number };  // NEW
}
```

### 2.2 createAutoSnapshot 新シグネチャ

```ts
export interface AutoSnapshotInput {
  added: readonly string[];
  modified: readonly string[];
  removed: readonly string[];
}

async createAutoSnapshot(input: AutoSnapshotInput): Promise<SnapshotMeta>
```

- `added` は source.readFile を呼ばない（metadata-only entry）
- `modified` / `removed` は remote から download して暗号化保存

### 2.3 rollback 挙動

| operation | rollback の挙動 |
|---|---|
| `added` | **uploader.unlink で remote から delete** |
| `modified` | snapshot から old content を restore（v0.9.x 同等） |
| `removed` | snapshot から old content を restore（新規実装） |

### 2.4 schema 1 互換（read-only）

- v0.9.x で作った snapshot は in-memory で schema 2 にマイグレートして read
- 全ファイルを `operation: 'modified'` として扱う（v0.9.x 同等挙動）
- **disk 上の manifest.enc は書き換えない**

## 4. Breaking change

| 項目 | v0.9.x | v0.10.0 |
|---|---|---|
| `SNAPSHOT_SCHEMA` | 1 | 2（新規作成は 2 のみ） |
| `SnapshotFileMeta` | path/storedName/size/sha 必須 | `operation` 追加、`storedName` 以下は `\| null` |
| `createAutoSnapshot` | `(files: string[])` | `(input: AutoSnapshotInput)` |
| `RollbackFileStatus` | 4 値 | 6 値（`deleted`, `skipped-no-delete` 追加） |

### 互換性マトリクス

| シナリオ | 挙動 |
|---|---|
| v0.10.0 で v0.9.x snapshot から rollback | ✅ schema 1→2 in-memory migration、modified-only として動作 |
| v0.10.0 で v0.10.0 snapshot から rollback | ✅ 新挙動（added=delete / modified=restore / removed=restore） |
| v0.9.x で v0.10.0 snapshot から rollback | ❌ `Unsupported snapshot schema: 2` で fail（downgrade 非サポート） |

## 5. 安全 invariant への影響

すべて維持：

- 暗号化必須（added は metadata-only なので暗号化対象外、modified/removed は AES-256-GCM）
- hard-exclude（`wp-config.php` 等）は **operation を問わず** rollback でスキップ（added の誤判定 delete を防ぐ）
- prepare/confirm gate 維持（MCP の `aiftp_rollback_prepare` は新 status `deleted` をプレビューに含める）
- production gate 維持
- 🆕 「every push is reversible」を **真に満たす**

## 7. Codex 独立レビュー観点

### 7.1 Data loss リスク（CRITICAL）

- [ ] added rollback の delete 判断は schema 2 の `operation` フィールド由来か（推測ヒューリスティックでないか）
- [ ] schema 1 snapshot を `operation='added'` と誤判定するロジックがないか
- [ ] hard-exclude チェックは `operation` よりも**先**に走るか

### 7.2 不可逆操作（HIGH）

- [ ] dryRun プレビューに `status='deleted'` 候補が明示されるか
- [ ] MCP `aiftp_rollback_prepare` に「delete されるファイル数」が入るか
- [ ] CLI 警告メッセージは「N files will be DELETED from remote」と明示か

### 7.3 Migration safety（HIGH）

- [ ] schema 1 → 2 の in-memory migration が idempotent か
- [ ] schema 1 snapshot の manifest.enc を**書き換えない**ことを test で保証しているか
- [ ] 未知 schema（3 以上）を読んだら明確に fail するか

### 7.4 Test coverage の盲点

- [ ] added-only push で planned が hard-exclude のみだった場合の挙動
- [ ] removed entry が hard-exclude の場合の挙動
- [ ] uploader.unlink が delete 中エラーした場合の partial state 扱い

## 8. A-7 試用期間中の検証手順

> 解約締切: Lolipop 2026-05-30 / Xserver 2026-05-31 / Sakura 2026-06-04

各サーバで 5 シナリオ実施（合計 15 ケース）：

1. **added-only rollback** ⭐ 最重要：local で新規ファイル作成 → push → rollback で **remote から消えている**
2. **modified rollback**（回帰テスト）
3. **mixed rollback**（added + modified を 1 push）
4. **schema 1 compat**：v0.9.2 で作った snapshot から rollback
5. **hard-exclude 保護**：擬似的に hard-exclude されたファイルが delete されない
